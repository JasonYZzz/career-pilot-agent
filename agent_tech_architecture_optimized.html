多 Agent 课件生成平台技术架构设计方案

面向生产落地的新版架构：以 Planner 控制面为核心，结合多智能体协作、RAG 证据链、MCP 工具生态、权限治理、评测回放与交付闭环，支撑从教学意图到可发布课件的全过程。

设计取向

不把多 Agent 做成“多个提示词串联”，而是做成可路由、可暂停、可评估、可追踪、可授权、可局部修复的工程系统。页面基于 2026-07-07 检索到的官方资料和工程经验整理。

一、设计原则

新版架构把“生成体验”与“工程可靠性”放在同一优先级：用户看到的是课件生成，系统内部真正运行的是有权限、有状态、有证据、有评测的任务编排。

Control Plane

Planner 是控制面，不是普通 Agent

Planner 负责意图识别、任务图、依赖、预算、并发、授权、handoff 和终止条件。内容生成 Agent 只负责专业产出，不能随意改工作流。

Evidence First

先证据、后生成

凡涉及教材、知识点、公式、事实、题目答案和引用，都先构建 Evidence Pack，再交给内容 Agent，避免无证据生成。

Least Privilege

工具权限最小化

MCP、导出、写库、发起联网检索、覆盖文件、发布课件等动作必须经过工具级 scope、策略校验和必要的人审。

Repair Loop

局部修复优先

评审失败后不全量推翻，而是定位到页面、组件、知识点、引用或交互状态，使用 checkpoint 进行局部重生。

二、优化后的技术架构

整体采用三层视角：控制面负责“该怎么做”，执行面负责“把事情做完”，治理面负责“能不能做、是否合格、如何追踪”。

1

用户入口与任务通道

统一承接对话、文件、模板、历史课件、教研规范和交付格式。

Chat / Workspace自然语言输入、模板选择、多轮澄清。
Upload GatewayPDF、Word、PPT、图片、音视频、题库。
Artifact Console预览、编辑、导出、版本和发布。
2

意图理解与上下文组装

将用户输入标准化为任务信号和约束。

Intent Classifier生成、改写、补证据、互动设计、导出、评估。
Slot Extractor年级、学科、知识点、教材版本、课时、风格。
Context Assembler历史会话、上传资料摘要、用户偏好、权限。
3

Multi-Agent Planner 控制面

核心优化点：Planner 输出 Typed Plan，而不是让大模型自由调用所有工具。计划必须包含 DAG、Agent 契约、工具 scope、预算、评审规则和失败回退。

Plan Graph Builder构建任务 DAG、依赖、可并发节点和阻塞点。
Policy-aware Router根据权限、风险、成本、延迟选择链路。
Execution Arbiter决定 agents-as-tools、handoff、parallel fan-out、human checkpoint。
Convergence Judge判断是否完成、重试、降级、局部修复或请求澄清。
4

专业 Agent 执行面

每个 Agent 拥有窄职责、独立上下文、明确输入输出 Schema 和工具白名单。

Teaching Designer教学目标、结构、活动、评价。
Knowledge RetrieverRAG / GraphRAG、证据包、引用。

Renderer / PackagerHTML、PPTX、JSON、资源打包。
5

交付与用户反馈闭环

交付物既是结果，也是下一轮改写和评估的输入。

Preview Runtime真实渲染、互动检查、移动端适配。
Export ServiceHTML、PPTX、教案、讲稿、练习、引用包。

知识与证据层

面向教学场景的 RAG 不只是向量召回，还要处理教材结构、章节层级、题目答案、公式、图片和教学目标。

Parser + OCR + Layout版面、表格、公式、图片、章节结构。

7

工具 / MCP / Skill 层

工具统一注册、统一授权、统一审计。对外部工具采用 MCP 连接，对内部能力采用函数工具或服务接口。

8

权限与安全治理

在每个边界进行判定：用户能否发起、Agent 能否访问、工具能否执行、产物能否发布。

RBAC / ABAC角色、租户、学科、数据域、动作粒度。
Approval Gate发布、覆盖、联网、写库、外发、敏感工具。
Data Policy隐私脱敏、版权标注、未成年人保护、保留周期。
9

Agent Runtime 与状态层

承载长任务、并发执行、可恢复状态和可观测性。

Run State / Checkpoint每个节点状态、输入输出、失败原因、恢复点。

10

正式交付前必须通过教学、事实、引用、交互、格式和安全检查。

11

平台基础设施

支撑模型网关、队列调度、对象存储、向量库、图数据库、缓存、日志、CI/CD、灰度发布和多租户隔离。

三、Multi-Agent Planner 编排逻辑优化

Planner 的关键优化是“确定性工作流 + Agentic 决策”的混合：确定性部分保证边界和稳定，Agentic 部分处理开放式教学设计。

1Task Intake归一化输入，识别请求类型、上下文、资料、目标格式和关键缺失字段。
2Risk & Permission Preflight在生成前检查租户、角色、数据域、工具 scope、联网许可和敏感动作。
3Plan DAG输出可执行任务图：节点、依赖、并发组、预算、截止条件和 checkpoint。
4Context Routing每个 Agent 只拿必要上下文：摘要、证据包、用户偏好、约束，不共享无关历史。
5Execution Strategy选择 agents-as-tools、handoff、router fan-out、skills 或自定义工作流。
6Convergence & Critic对阶段产物做教学、事实、引用、交互和格式评审，决定通过或局部返工。
7Delivery & Learning交付产物、记录 trace、保存可复用记忆和失败样例，进入评测集。

四、Planner 的编排模式选择

结合最新资料和工程经验，建议不要单押一种多 Agent 模式，而是由 Planner 按任务形态动态选择。

编排模式 适用场景 优势 风险 本平台建议
Agents as Tools Supervisor 保持最终答复所有权，调用专家 Agent 完成边界清晰的子任务。 控制集中、上下文隔离、评审统一，适合正式课件生成主链路。 多一次汇总调用，延迟和成本略高。 默认模式。教学设计、内容生成、检索、渲染作为有契约的工具型 Agent。
Handoff 某个专家需要接管多轮对话，例如需求澄清、人工编辑、复杂交互配置。 减少重复路由，适合连续多轮专项工作。 上下文交接容易膨胀或丢消息，权限边界需要重新校验。 只用于需要专家接管用户交互的分支，并传递过滤后的 handoff context。
Router Fan-out 多个相互独立的分析任务可并发执行，例如资料摘要、题库抽取、案例检索。 延迟低、适合大上下文多域任务。 结果合并复杂，重复检索可能增加成本。 用于 RAG、质量检查、多格式导出和并行专家评审。
Skills / Prompt Packs 同一 Agent 需要按学科、学段、互动组件加载专用知识。 少建 Agent，灵活，适合提示词和模板复用。 上下文积累过多会影响成本和准确性。 用于学科风格、模板规则、互动组件指南，按需加载并限制 token 预算。
Custom Workflow 有明确业务状态机、审批、回滚、评估门禁和发布流程。 可测试、可回放、可恢复，最适合生产系统。 开发复杂度高，需要状态模型和运维能力。 作为平台主干；Agent 决策嵌入节点，不让模型直接掌控全部流程。

五、权限与治理设计

多 Agent 系统的权限问题比传统应用更复杂：不是只有用户访问控制，还要控制 Agent 可见上下文、工具可执行动作和产物可发布范围。

用户权限RBAC 控制角色，ABAC 控制学科、学校、租户、资料来源、课程包和操作范围。
Agent 权限每个 Agent 使用最小上下文、最小工具集和最小数据域；禁止跨租户读取记忆或素材。
工具权限Tool Registry 记录 scope、风险等级、幂等性和审批规则；高风险工具必须 human approval。
MCP 授权远程 MCP 按 OAuth 2.1 资源服务器模型校验 token、audience、scope 和 resource metadata。
数据权限上传资料、教材、题库、学生信息、历史课件分别设数据域和保留策略，Evidence Pack 附带来源限制。
发布权限下载、分享、发布、覆盖旧版本、联网引用和外发文件都作为独立动作审计。

六、路由策略与失败回退

路由策略应显式编码，不应完全依赖模型自由判断。下面给出推荐的混合编排逻辑。

Planner Routing Policytyped plan
function buildPlan(input, user, session) {
  normalized = normalize(input, session)
  intent = classifyIntent(normalized)
  slots = extractRequiredSlots(normalized)
  risk = assessRisk(normalized, user)
  permission = authorizePreflight(user, intent, normalized.resources)

  if (!permission.allowed) return stop(permission.reason)
  if (slots.criticalMissing) return askClarification(slots.missing)

  plan = new PlanDAG({
    budget: estimateBudget(intent, normalized.files),
    traceLevel: risk.high ? "full" : "standard",
    checkpoint: true
  })

  if (intent.needsEvidence) {
    plan.add("parse_files")
    plan.add("retrieve_evidence", dependsOn: ["parse_files"])
    plan.add("build_evidence_pack", dependsOn: ["retrieve_evidence"])
  }

  plan.add("teaching_design", needs: ["intent", "constraints"])
  plan.add("content_generation", needs: ["teaching_design", "evidence_pack"])
  plan.addParallel(["interaction_design", "visual_design", "quiz_design"])
  plan.add("render_artifact")
  plan.add("evaluate_artifact")

  if (risk.requiresApproval) plan.add("human_checkpoint")
  plan.add("deliver_or_repair")
  return plan
}
Execution Arbitermode selection
function selectExecutionMode(task) {
  if (task.hasDeterministicStateMachine) {
    return "custom_workflow"
  }

  if (task.requiresUserFacingSpecialistDialog) {
    return "handoff_with_filtered_context"
  }

  if (task.isBoundedSpecialistWork) {
    return "agent_as_tool"
  }

  if (task.canRunIndependentBranches) {
    return "router_parallel_fanout"
  }

  if (task.onlyNeedsDomainPromptOrTemplate) {
    return "skill_loaded_single_agent"
  }

  return "supervisor_managed_workflow"
}

七、运行时闭环

运行时重点不是“能生成一次”，而是能长任务执行、失败恢复、回放问题、评估退化和持续优化。

主执行链路

适用于从零生成完整课件。

Intake接收用户目标、资料、模板、输出格式和权限上下文。
Preflight权限、风险、缺失字段、资料可解析性和预算预估。
Plan DAG确定节点、依赖、并发组、checkpoint 和工具 scope。
ExecuteSupervisor 调用专业 Agent 和工具，所有节点写入 trace。
Evaluate事实、引用、互动、版式、可访问性、安全和交付格式检查。
Repair失败节点局部重试，必要时请求用户确认或降级。
Deliver生成 HTML/PPTX/教案/练习，保存版本、引用包和运行摘要。

可观测与评测链路

适用于线上质量治理。

Trace记录模型调用、工具调用、handoff、guardrail、审批和 custom span。
Metrics统计成功率、首 token 延迟、总耗时、成本、重试率、人工介入率。
Golden Set沉淀典型学科、题型、互动组件和失败案例。
Replay用历史 trace 回放新版本，检查 planner、prompt、模型或工具变更影响。
A/B灰度比较不同 planner 策略、检索策略、模型路由和评审门槛。
Distill把高频、稳定的 Agent 行为沉淀为模板、skill 或 deterministic node。

八、落地路线

建议分三期交付，先打通闭环，再提升智能度，最后做规模化治理。

阶段 目标 关键建设 验收标准
Phase 1：MVP 闭环 让用户能从资料生成可预览课件。 Intent Router、Plan DAG、3 个核心 Agent、基础 RAG、HTML/PPTX 导出、Trace。 能生成、能预览、能导出、失败可定位，不追求复杂互动全覆盖。
Phase 2：质量闭环 提升事实、引用、互动和版式稳定性。 Evidence Pack、Evaluator、局部修复、Golden Set、Trace Replay、权限审批。 典型样例回归通过；错误能被评审发现并局部修复。
Phase 3：规模化生产 支撑多学科、多租户、多模型、多工具稳定上线。 MCP Gateway、OAuth/scope、模型网关、成本优化、灰度发布、运营看板。 可灰度、可审计、可回滚、可扩展，核心指标可持续监控。
